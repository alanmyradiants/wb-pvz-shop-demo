# wb-pvz-shop-demo

Демо-магазин с выбором пункта выдачи Wildberries в качестве адреса доставки.

**Идея:** использовать сеть из 32 000+ ПВЗ Wildberries как точки выдачи для заказов своего магазина (в Telegram-боте, на сайте и т.д.). Покупатель оформляет заказ и выбирает любой удобный ПВЗ Wildberries по всей России. Продавец затем вручную оформляет отправление через [WB Track](https://track.wildberries.ru/) на ID выбранной точки.

## Демо

После публикации репозитория откройте через GitHub Pages:

> https://alanmyradiants.github.io/wb-pvz-shop-demo/

(Чтобы включить Pages: Settings → Pages → Source: «Deploy from a branch» → `main` / root → Save.)

## Что внутри

- [`index.html`](index.html) — однофайловый магазин-демо. Сначала пытается тянуть данные ПВЗ напрямую с сервера WB, при недоступности (CORS) — лениво подгружает файлы из `data/`.
- [`scripts/parse_wb_pvz.py`](scripts/parse_wb_pvz.py) — Python-скрипт генерации локального кэша. Скачивает базу с WB, парсит адреса, делит на бакеты до 100 КБ, кладёт в `data/`. Без зависимостей.
- `data/` — сгенерированный сплит-кэш базы ПВЗ. Содержит:
  - `index.js` (~310 КБ) — индекс городов с привязкой к бакет-файлам;
  - `city-Москва-1.js`, `city-Москва-2.js`, …, `city-Санкт-Петербург-1.js`, … — крупные города в своих файлах (с чанками если очень большие);
  - `letter-А-1.js`, `letter-Б-1.js`, … — мелкие города сгруппированы по первой букве;
  - всего ~70 файлов по ≤100 КБ каждый.

Не лежит в репозитории по умолчанию — генерируется скриптом, нужен только как фолбэк.

## Как запустить

```bash
git clone https://github.com/alanmyradiants/wb-pvz-shop-demo.git
cd wb-pvz-shop-demo
# Откройте index.html в браузере (двойной клик)
```

Магазин при загрузке сначала пытается скачать список ПВЗ напрямую с сервера WB — на момент написания CORS у их CDN открыт, и это работает из коробки. В шапке появится зелёный бейдж «🟢 живые данные WB».

Если хотите офлайн-версию или CORS вдруг закроют — сгенерируйте локальный кэш (см. ниже).

## Как работает выбор ПВЗ

1. Покупатель добавляет товары в корзину.
2. На шаге оформления вводит имя и телефон.
3. В поле **Город** начинает печатать — браузер показывает подсказки из реальной базы (топ-500 городов).
4. После выбора города дропдаун **Пункт выдачи Wildberries** заполняется всеми ПВЗ этого города. В lazy-режиме нужный бакет-файл подгружается на лету (1 запрос ≤100 КБ).
5. После выбора точки видны её ID, координаты и адрес. **ID — внутренний идентификатор Wildberries**, по которому продавец дальше оформляет WB Track.

## Источник данных

Все ПВЗ берутся из публичного эндпоинта Wildberries:

```
https://static-basket-01.wb.ru/vol0/data/all-poo-fr-v8.json
```

Это статический файл на CDN, отдаёт около 4 МБ JSON со всеми точками всех стран. Авторизация не требуется.

В шапке магазина бейдж показывает текущий источник:

- 🟢 **живые данные WB** — данные только что загружены с сервера WB;
- 📁 **локальный кэш (lazy)** — используется сплит-кэш в `data/`.

## Архитектура сплит-кэша

Зачем разбивать 5 МБ на 70 файлов? Чтобы:

1. **Каждый файл влезал в типичный лимит размера для git/CDN/CI**. Один большой 5-мегабайтный JS пушить через API больно, drag-and-drop в GitHub веб-UI тоже капризничает.
2. **Браузер грузил только нужное**. При выборе «Уфа» подтягивается ~50 КБ вместо всех 5 МБ. Стартовый индекс (~310 КБ) загружается один раз.
3. **Кэширование умнее**: точки маленьких городов меняются редко, точки Москвы обновляются часто — теперь это разные файлы, и CDN/браузер кэшируют их независимо.

Логика разбиения:
- Города с ≥300 точек — каждый в свой `city-{name}-{N}.js` (Москва ≈3800 точек → 6 чанков).
- Все остальные — группируются по первой букве, упаковываются в `letter-{Б}-{N}.js` до лимита 100 КБ.
- В `data/index.js` лежит индекс `[city, [bucket_ids], count]` — чтобы по выбранному городу понять, какие файлы загружать.

## Генерация локального кэша

```bash
python scripts/parse_wb_pvz.py
```

Скрипт скачивает актуальный JSON, парсит адреса (поддерживает 4 формата WB), группирует точки и сохраняет результат в `data/`. Зависимостей нет — только стандартная библиотека Python 3.

После этого магазин будет работать даже без интернета (live-фолбэк включится автоматически).

## Что это НЕ делает

- **Не оформляет отправление в WB Track автоматически** — для этого продавец вручную в приложении WB вбивает ID выбранной точки. Публичного API WB Track для бизнеса пока нет.
- **Не принимает оплату** — это демо. В реальном магазине здесь интеграция с эквайрингом.
- **Не учитывает статус ПВЗ** (недоступен, возможны очереди) — поле есть в выгрузке, но в демо не показано.
