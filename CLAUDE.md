# wb-pvz-shop-demo

Демо-магазин с выбором пункта выдачи Wildberries в качестве адреса доставки. Идея — использовать сеть 32 000+ ПВЗ WB как точки выдачи для своего магазина.

## Стек
- **Frontend:** один HTML-файл `index.html` (без сборщиков)
- **Data prep:** Python скрипт `scripts/parse_wb_pvz.py` (без зависимостей, только стандартная либа)
- **Хостинг:** GitHub Pages
- **GitHub:** https://github.com/alanmyradiants/wb-pvz-shop-demo (публичный)
- **Demo:** https://alanmyradiants.github.io/wb-pvz-shop-demo/

## Что внутри
- `index.html` — однофайловый магазин. Сначала тянет ПВЗ напрямую с WB, при CORS — фолбэк на `data/`
- `scripts/parse_wb_pvz.py` — генерация локального сплит-кэша
- `data/` — сплит-кэш ~70 файлов до 100 КБ каждый (генерится скриптом, не лежит в репо)

## Источник данных
Публичный WB-эндпоинт без авторизации:
```
https://static-basket-01.wb.ru/vol0/data/all-poo-fr-v8.json
```
~4 МБ JSON всех точек всех стран.

## Архитектура сплит-кэша
- Города ≥300 точек → свой файл `city-{name}-{N}.js` (Москва ≈3800 → 6 чанков)
- Мелкие города → группа по первой букве `letter-{Б}-{N}.js` до 100 КБ
- `data/index.js` (~310 КБ) — индекс городов → бакеты

---

## Важно знать (что НЕ делает)

- **Не оформляет WB Track автоматически** — публичного API для бизнеса нет. Продавец вручную вбивает ID ПВЗ
- **Не принимает оплату** — это демо. В проде здесь эквайринг
- **Не учитывает статус ПВЗ** (закрыт, очереди) — поле есть в выгрузке, но в демо не используется

## Если CORS на WB закроют
Включить локальный кэш:
```bash
python scripts/parse_wb_pvz.py
```
Магазин автоматом переключится на режим «локальный кэш (lazy)» (бейдж в шапке).
